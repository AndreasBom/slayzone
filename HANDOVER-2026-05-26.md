## What works now

- **`-L slayzone` socket cutover** — every ssh-emitted tmux command
  runs through `~/.slayzone/bin/slz-tmux` (installed by
  `remote-hook-installer`) which pins `-L slayzone -f ~/.tmux/slayzone.conf`
  on every invocation. User's default-socket tmux is untouched.
- **Session tagging** — every spawned tmux session is tagged with
  `@slz-task-id`, `@slz-tab-id`, `@slz-instance-id`, `@slz-mode`,
  `@slz-created` via `set-option` calls in the inner script *before*
  the bash exec.
- **Instance id** persists at `${userData}/.slayzone-instance-id`
  (UUID, env override via `SLAYZONE_INSTANCE_ID`). Exposed via
  `window.api.instance.getId()`.
- **tmux-resurrect + tmux-continuum installed** on remote (best-effort;
  failure falls back gracefully — wrapper + conf are essential, plugins
  optional). Auto-restore is intentionally OFF; user invokes
  `prefix + Ctrl-R` to resurrect after reboot. Saving runs every 15 min.
- **Kill IPC** — three IPC handlers in `apps/app/src/main/tmux-handlers.ts`:
  `tmux:listSessions`, `tmux:killSession(projectId, sessionName, mode)`,
  `tmux:killAllSessions(projectId, scope)`. Probes both slayzone and
  default sockets (legacy migration).
- **Remote Sessions panel** in Project Settings → Environment (when
  `execution_context.type === 'ssh'`). Per-row kill with two-click
  confirm, "Kill all owned by this SlayZone", "Kill all (advanced)"
  behind toggle, legacy group with explicit "no metadata" copy.
- **Tab context menu "Kill Remote Session"** — right-click on terminal
  tab when project is ssh. Calls `tmux:killSession` with mode.
- **Tagged kill clears `provider_config[mode].conversationId`** via
  `setProviderConversationId(cfg, mode, null)`. Legacy untagged kill
  is kill-only.
- **`SessionEnd` auto-clear** — when claude/codex/qwen-code emits
  `SessionEnd` (clean `/exit`), `agent-hook.ts` nulls the stored
  conversation id. `Stop` is *not* touched (fires every turn).
- **Resume failure fallback** — `claude-adapter.ts`, `codex-adapter.ts`,
  `qwen-adapter.ts` all emit `code: 'SESSION_NOT_FOUND'` via
  `detectError` on their invalid-session strings. `pty-manager.ts`
  broadcasts `pty:session-not-found`, renderer's `handleSessionInvalid`
  clears the id, kills, respawns fresh. New diagnostic events:
  `pty.resume_failed` + `pty.resume_fallback_to_fresh`.
- **`waitForShutdown(sessionId, timeoutMs)`** helper for callers that
  need to await PTY finalize (used by kill IPC).
- **`ensureRemoteWrapper`** bootstrap — Step 2 IPC handlers probe both
  wrapper + slayzone.conf, full re-install if either missing.

## Where to start the worktree feature

Existing infrastructure to lean on:

- `task.worktree_path TEXT` column already exists; renderer already
  reads it for cwd resolution
  (`TaskDetailPage.tsx:effectiveRepoPath`).
- `createWorktree(repoPath, wtPath, branchName)` in
  `packages/domains/worktrees/src/main/git-worktree.ts:110` — works
  against local fs. For ssh, the equivalent needs to run remotely via
  `runGit` (`packages/domains/worktrees/src/main/run-git.ts`) which
  already routes through ssh when `execution_context.type === 'ssh'`.
- `resolveProjectExecutionContext(db, projectId)` returns the target
  + workdir + shell for ssh projects (already used by
  `apps/app/src/main/tmux-handlers.ts`).
- The kill flow in tmux-handlers.ts:`tmux:killSession` is where the
  diff-prompt + worktree-remove confirmation needs to land.

Order of work:

1. **Schema/state check** — confirm `task.worktree_path` is per-task
   (it is; one column on tasks).
2. **Auto-create on task-create for ssh projects** — hook into the
   task-create IPC handler in `packages/domains/task/src/main/`.
   For host projects, call `createWorktree` directly. For ssh, build
   an analogous `createWorktreeRemote(target, repoRemotePath, wtRemotePath, branch)`
   that runs `git worktree add` via the existing `runGit` transport.
3. **Diff fetch on kill** — extend `tmux:killSession` (or a new
   `tmux:killSessionWithWorktree`) to first run
   `git -C <worktree_path> diff --stat` + `git status --porcelain`
   over ssh, return the result to the renderer.
4. **Renderer confirm dialog** — surface the diff, let user pick:
   - "Kill + stash" (creates `git stash` then removes worktree)
   - "Kill + commit" (opens commit message input, commits, removes worktree)
   - "Kill + discard" (`git checkout .` + `git clean -fd` + remove worktree)
   - "Cancel"
   Plus a "no changes — just kill" fast path when porcelain is empty.
5. **Worktree remove** — `git worktree remove <path>` over ssh,
   followed by deleting the branch if it was task-owned.

## Risk areas to test before merge

- **Task created on host project, then `execution_context` swapped
  to ssh**: the existing `worktree_path` was a host path; new ssh
  spawn will fail. Decide: clear or migrate.
- **User deletes a task without killing the session first**: who
  owns the worktree cleanup? Probably `task:archive`/`task:delete`
  handlers need the same diff-prompt flow.
- **Concurrent kills**: two tabs in the same task → kill one, both
  see worktree disappear. Plan v6's per-task scope handles this —
  worktree remove only on LAST tab kill, or on explicit "Delete task"
  action. Open design question.
- **`git worktree remove` refuses on dirty wd** — flow must `--force`
  ONLY after diff-prompt and user choice. Never default to `--force`.

## Things that look wrong but are intentional

- `apps/app/src/main/tmux-handlers.ts` imports from
  `@slayzone/worktrees/main` and `@slayzone/task/shared` — looks like
  cross-domain coupling. It's intentional: the file lives in the
  composition root precisely so it can pull from any domain without
  creating cycles.
- `RemoteSessionsPanel` lives in `projects/client/` not
  `worktrees/client/`. Same reason — projects domain owns Project
  Settings; would be a cycle if it imported from worktrees.
- `Stop` is NOT a clear-conversation trigger. Only `SessionEnd`.
  This was the v2 critical bug — `Stop` fires every turn boundary.
- Auto-restore (`@continuum-restore 'on'`) is OFF. User runs
  `prefix + Ctrl-R` manually after reboot. Otherwise continuum would
  resurrect zombie sessions the user deleted via the Step 2 panel.

## Quick repro commands for the user's setup

- SlayZone dev: `SLAYZONE_DEVTOOLS_PORT=9222 pnpm dev`
- Force re-install on a remote (e.g. if remote files got into a
  weird state):
  `ssh <target> 'rm -f ~/.slayzone/migration-done.v1; rm -rf ~/.tmux/plugins'`
  → next PTY spawn re-runs the whole install.
- See diagnostics events live (Python on host):
  ```sh
  python -c "
  import os, sqlite3, json
  db = os.path.join(os.environ['APPDATA'], 'slayzone', 'slayzone.dev.diagnostics.sqlite')
  con = sqlite3.connect(db)
  con.row_factory = sqlite3.Row
  cur = con.execute(\"SELECT ts_ms, event, payload_json FROM diagnostics_events WHERE event LIKE 'pty.tmux_%' OR event LIKE 'pty.resume_%' ORDER BY ts_ms DESC LIMIT 20\")
  for r in cur.fetchall():
      print('---', r['ts_ms'], r['event'])
      print(json.dumps(json.loads(r['payload_json']) if r['payload_json'] else None, indent=2)[:800])
  "
  ```
